# CTRG Presentation Script For Beginner Presenter

## Title
CTRG Two-Stage Research Grant Review Management System

## Who This Script Is For
This version is written for a beginner presenter.
It is designed to help you explain the system clearly to a professor, even if the professor has not used the software before.

## Suggested Time
8 to 10 minutes

## How To Present

When you present:

- speak slowly
- pause after each major point
- do not try to explain every technical detail at once
- first explain the problem
- then explain the users
- then explain the workflow
- then explain the technical side

If you forget a line, just explain the idea in your own words. That is completely fine.

## Full Presentation Script

## 1. Opening

Good morning sir.

Today I will present our project, which is called the CTRG Two-Stage Research Grant Review Management System.

This project is a web-based platform that helps manage the full process of research grant proposals. Instead of handling proposals through emails, printed files, and manual follow-up, this system brings everything into one place.

The main purpose of this project is to make the research grant review process more organized, transparent, and easy to manage for all people involved.

---

## 2. What Problem Does This Project Solve

Before explaining the system, I want to explain the problem.

In many universities, research grant proposals are managed manually. A principal investigator submits a proposal, then the authority has to send it to reviewers, collect feedback, request revision if needed, and finally make a decision.

When this is done manually, several problems happen.

First, it becomes difficult to track which proposal is in which stage.

Second, communication becomes slow because everything depends on email and manual follow-up.

Third, reviewers may be overloaded, while some reviewers may have no work.

Fourth, deadlines can be missed easily.

And finally, it becomes hard to keep a proper record of who made which decision and why.

So our project solves these problems by creating a centralized digital system for the complete grant review process.

---

## 3. Main Idea Of The System

The main idea of this project is very simple.

A proposal does not just get submitted and accepted directly.

Instead, it moves through a structured review process with two stages.

In Stage 1, reviewers evaluate the original proposal.

Then the SRC Chair studies the reviews and can make one of three decisions:

- reject the proposal
- accept the proposal without correction
- or tentatively accept it and ask for revision

If revision is requested, the PI submits a revised proposal.

Then in Stage 2, reviewers check whether the PI addressed the previous comments properly.

After that, the SRC Chair makes the final decision.

So this system is not only for uploading files. It is a full workflow management system.

---

## 4. Users Of The System

This system has three main users.

### Principal Investigator or PI

The PI is the person who submits the research proposal.

The PI can:

- create an account
- submit a proposal
- see proposal status
- receive review feedback
- upload revised proposal if needed
- see the final decision

So the PI side is mainly for submission, tracking, and revision.

### Reviewer

The reviewer is responsible for evaluating the proposal.

The reviewer can:

- receive assigned proposals
- open review forms
- submit Stage 1 review
- submit Stage 2 review
- save draft review before final submission
- see review deadlines

So the reviewer side is focused on evaluation.

### SRC Chair

The SRC Chair is the administrator of the process.

The SRC Chair can:

- create and manage grant cycles
- manage PI and reviewer accounts
- approve PI accounts
- approve submitted proposals for review
- assign reviewers
- monitor submitted reviews
- reject a review from a particular reviewer if needed
- send a review back to the reviewer again
- request revision from the PI
- make Stage 1 and final decisions
- generate reports

So the SRC Chair controls the whole workflow.

---

## 5. How The Full Workflow Works

Now I will explain the complete workflow step by step.

### Step 1. PI Registration

First, the PI creates an account in the system.

After registration, the PI account remains inactive until SRC Chair approval.

This is important because the authority should verify who is entering the system.

### Step 2. SRC Chair Approves PI Account

The SRC Chair can see PI accounts in the PI management section.

Then the SRC Chair activates the PI account.

After approval, the PI can log in and use the system.

### Step 3. Proposal Submission

After logging in, the PI creates and submits a proposal.

The proposal contains:

- title
- abstract
- PI information
- funding request
- proposal file
- keywords and research area

Once submitted, the proposal appears in the SRC Chair dashboard.

### Step 4. SRC Chair Approves Proposal For Review

After submission, the SRC Chair reviews the submitted proposal list.

Then the SRC Chair approves the proposal for Stage 1 review by assigning reviewers.

At this point, the proposal moves from submitted state into Stage 1 review.

### Step 5. Reviewer Assignment

The SRC Chair selects one or more reviewers for the proposal.

This system supports multiple reviewers for the same proposal.

This is important because a research proposal usually should not depend on only one opinion.

Also, the system checks reviewer workload so the chair can avoid assigning too many proposals to the same reviewer.

### Step 6. Stage 1 Review

In Stage 1, the reviewer evaluates the original proposal using eight scoring criteria.

These include:

- originality
- clarity
- literature review
- methodology
- impact
- publication potential
- budget appropriateness
- timeline practicality

The total score is out of 100.

The reviewer also writes narrative comments and gives a recommendation.

### Step 7. SRC Chair Reviews Submitted Reviews

After reviewers submit their reports, the SRC Chair can open the combined review view.

There, the chair can see:

- which reviewer submitted which review
- the reviewer’s score
- comments
- recommendation

If needed, the chair can reject a particular reviewer’s review with a comment.

The chair can also send the same review back to that reviewer again for correction or re-review.

This means the chair has control over review quality, not just final decisions.

### Step 8. Stage 1 Decision

After considering the Stage 1 reviews, the SRC Chair makes one of three decisions:

- reject
- accept without correction
- tentatively accept

If rejected, the process ends.

If accepted without correction, the proposal can move directly toward final acceptance.

If tentatively accepted, the PI must revise the proposal.

### Step 9. Revision Submission

If revision is requested, the PI sees that status on the dashboard.

Then the PI uploads:

- revised proposal
- optional response to reviewers

The system also tracks revision deadline.

### Step 10. Stage 2 Review

After revision submission, the SRC Chair assigns Stage 2 reviewers.

In Stage 2, the reviewer does not evaluate the whole proposal from the beginning.

Instead, the reviewer checks whether the PI addressed the earlier concerns.

Then the reviewer gives a revised recommendation.

### Step 11. Final Decision

Finally, the SRC Chair makes the final decision.

This can be:

- final accepted
- or final rejected

If accepted, the approved grant amount and remarks can also be recorded.

So the system supports the full lifecycle from beginning to final decision.

## 6. What The PI Sees

The PI dashboard is designed to be simple.

The PI can see:

- all submitted proposals
- proposal status
- revision deadline
- action needed
- final decision

This helps the PI understand exactly what is happening without needing to contact the office repeatedly.

So from the PI side, the system improves visibility and communication.

## 7. What The Reviewer Sees

The reviewer dashboard shows assigned reviews.

The reviewer can see:

- proposal title
- proposal code
- deadline
- whether the review is pending or completed
- whether the chair sent it back again
- chair comments if a review was rejected or returned

So reviewers can clearly understand what work they need to do and by when.

---

## 8. What The SRC Chair Sees

The SRC Chair dashboard is the control center of the whole system.

The chair can see:

- total proposals
- pending reviews
- proposals waiting for decision
- proposals waiting for revision
- active grant cycle

The chair can also open:

- proposal management
- PI management
- reviewer management
- reports
- audit logs

So the SRC Chair can monitor and manage the whole grant process from one interface.

## 9. Important Features Of The System

Now I will briefly explain the important features.

### Role-based access

Each user sees only their own part of the system.

PIs cannot access chair functions.
Reviewers cannot manage users.
SRC Chair has administrative control.

### Two-stage review workflow

This is the core academic logic of the project.

It allows both initial evaluation and post-revision evaluation.

### Multiple reviewers per proposal

A proposal can be reviewed by more than one reviewer.

This improves fairness and evaluation quality.

### Review rejection and re-review by SRC Chair

If a review is poor or incomplete, the chair can reject it and send it back with comments.

This is a strong quality control feature.

### Deadline tracking and reminders

The system can monitor deadlines and send reminders automatically.

### Reporting

The system can generate combined review reports and summary reports.

### Audit trail

Important actions are stored for accountability.

### Secure file storage

Proposal files can be protected using encrypted storage configuration.

## 10. Technical Explanation In Simple Language

Now I will explain the technical side in a simple way.

This is a full-stack application.

The backend is built with Django and Django REST Framework.

The backend is responsible for:

- storing users
- storing proposals
- storing reviews
- handling permissions
- handling business logic
- handling status transitions
- sending notifications
- generating reports

The frontend is built with React and TypeScript.

The frontend is responsible for:

- user interface
- forms
- dashboards
- navigation
- API communication

The database stores the actual records such as users, proposals, assignments, decisions, and reviews.

So in simple words, the frontend is what the user sees, and the backend is the logic behind the system.

## Tips For Answering Professor Questions

If the professor asks, "Why is this project important?"

You can answer:

Because universities often handle grant review manually, which creates confusion and delay. Our system makes the process centralized and trackable.

If the professor asks, "Why two-stage review?"

You can answer:

Because some proposals are not rejected immediately. They need revision, so the second stage helps evaluate whether the PI improved the proposal after feedback.

If the professor asks, "Why multiple reviewers?"

You can answer:

Because relying on a single reviewer may not be fair. Multiple reviewers improve balance and quality of evaluation.

If the professor asks, "What makes this more than a simple CRUD project?"

You can answer:

Because it includes workflow logic, role-based permissions, proposal lifecycle tracking, review decisions, revision handling, and administrative control.
